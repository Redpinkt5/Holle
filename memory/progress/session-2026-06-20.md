# Session 2026-06-21（再续）

## 用户反馈

Pet 运行后，AI 说“已经播放”，但实际没有声音。用户问 LLM 是怎么调用代码里的播放功能的。

## 原因

Holle Music 的 LLM 调用播放有两种路径：

1. **Tool-calling 服务（DeepSeek/Ark）**：
   - LLM 收到系统提示 + 工具定义后，生成类似 `{"name": "play_song", "arguments": "{\"title\": \"十年\"}"}` 的 tool_call。
   - `DeepSeekService.chat()` / `ArkService.chat()` 把 tool_call 返回给调用方。
   - `Pet` 的 `ai_worker` 循环里调用 `tools.execute("play_song", args)`，最终执行 `_tool_play_song`。

2. **纯文本服务（MiniMax/OpenAICompatible）**：
   - 模型只返回文本，不会生成 tool_call。
   - 代码通过关键词检测播放意图，本地调用 `tools.execute("search_local", ...)` 和 `tools.execute("play_song", ...)`。

**实际没播放的根因**：`AITools._tool_play_song` 之前一直调用 `self._player._send_cmd(...)`，这只会把命令写到 `~/.holle_music/pet_cmd.json`，由主程序读取执行。当 Pet 单独运行（没有开 TUI）时，主程序不在，`pet_cmd.json` 没人处理，所以命令石沉大海。AI 以为工具执行成功（返回“正在播放”），但实际播放器没动。

## 修复

在 `PetPlayer` 新增三个方法，自动判断当前是 IPC 模式还是独立模式：

- `play_song(song)`：主程序在 → 发 IPC；不在 → 直接调用 `standalone_player.play(song)`。
- `set_volume_pct(volume)`：主程序在 → 发 IPC；不在 → 直接调 `standalone_player.set_volume()`。
- `set_mode(mode)`：主程序在 → 发 IPC；不在 → 直接调 `standalone_player.set_play_mode()`。

然后让 `AITools` 的 `play_song`、`set_volume`、`set_mode` 都改走这些公共方法。

# Session 2026-06-21（五续）

## 用户反馈

合并气泡后出现了死循环：正在播放的气泡一直重复打印歌单，停不下来。

## 原因

`show_now_playing` 用 `append=True` 合并到当前 AI 回复气泡。但 `on_song_end_check` 会在歌曲切换时不断调用 `show_now_playing` 来刷新封面，每次 append 都会把整段“正在播放 + 接下来列表”再追加一遍，于是内容越滚越多，形成死循环。

## 修复

把 `append` 改成 `merge` 模式：

1. `BubbleManager` 记住最近一次**纯 AI 回复**（`_last_ai_response`）。
2. 当 `show_now_playing` 用 `merge=True` 时，把新的“正在播放”内容与 `_last_ai_response` 拼接，但**不会更新 `_last_ai_response`**。
3. 下次刷新时，仍然用同一段 AI 回复 + 新的正在播放信息，因此不会重复追加。
